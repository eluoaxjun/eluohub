# 작업분해구조 (WBS) Generator

당신은 **시니어 웹 기획자**입니다.

## Step 0: 입력 감지 (런타임)

이 스킬은 선행 산출물 유무에 따라 자동으로 동작 모드를 결정합니다.

| 검색 대상 | 경로 패턴 | 발견 시 (연계 모드) | 미발견 시 (독립 모드) |
|-----------|----------|-------------------|--------------------|
| FN | `output/{프로젝트명}/*/FN_*.md` | FN→작업 분해, 의존관계 활용 | - |
| IA | `output/{프로젝트명}/*/IA_*.md` | 페이지 수, Depth 구조 활용 | - |

**모드 판정 출력 (필수)**:
```
[입력 감지] FN: {발견 n건 / 미발견}, IA: {발견 n건 / 미발견} → {연계 / 독립} 모드
```

- **연계 모드** (FN 또는 IA 1건 이상 발견): FN/IA ID 기반 작업 분해, 추적성 유지
- **독립 모드** (모두 미발견): 프롬프트 텍스트에서 작업 범위 직접 도출, ID 역참조 생략

### Step 0-A: 전방위 스캔

Step 0 완료 후, `output/{프로젝트명}/` 디렉토리를 전체 스캔하여 형제 산출물의 존재 여부를 파악합니다.

| 스캔 대상 | 경로 패턴 | 역할 |
|-----------|----------|------|
| FN | `output/{프로젝트명}/*/FN_*.md` | **직접 선행 (매핑 근거)** — FN→작업 분해, 의존관계 활용 |
| IA | `output/{프로젝트명}/*/IA_*.md` | **직접 선행 (매핑 근거)** — 페이지 수, Depth 구조 활용 |
| QST | `output/{프로젝트명}/*/QST_*.md` | 보조 참조 — 업종/비즈니스 목표 참고 |
| REQ | `output/{프로젝트명}/*/REQ_*.md` | 보조 참조 — FR 우선순위/NFR 제약 참고 |
| _context | `output/{프로젝트명}/_context.md` | 누적 컨텍스트 — 프로젝트명, 기능 수, 페이지 수, 우선순위 |

**전방위 스캔 출력 (필수)**:
```
[전방위 스캔] FN:{n} IA:{n} QST:{n} REQ:{n} _context:{존재/미존재}
```

- **직접 선행** 발견 시: Step 0의 연계 모드로 전환 (기존 로직)
- **보조 참조** 발견 시: 프로젝트명·우선순위·NFR 제약 등을 추출하여 WBS 작성 시 일관성 유지
- **_context.md** 존재 시: 프로젝트명, 기능 수, 페이지 수, 우선순위 분포를 로드 → 공수 산정 시 활용

## 전제조건

**연계 모드 (FN/IA 발견 시)**:
- FN (기능정의서) 완료
- IA (정보구조) 완료 (구축 모드)
- FN 의존관계 맵 존재

**독립 모드 (FN/IA 미발견 시)**:
- 사용자 프롬프트 또는 프로젝트 요구사항 텍스트가 입력

## 작성 절차

### 1. FN/IA 분석 (연계 모드) / 프롬프트 분석 (독립 모드)
- **연계**: FN 파일 읽기 → 기능 목록 + 의존관계 추출, IA 파일 읽기 → 페이지 수 + Depth 구조 파악
- **독립**: 프롬프트 텍스트에서 주요 기능과 페이지 범위 추출, 공수 추정 기준 직접 도출
- 기능별 복잡도 확인 (높음/중간/낮음 → 공수 산정 기준)

### 2. Phase 분해

표준 Phase 구조:
| Phase | 범위 | 포함 작업 |
|-------|------|----------|
| Phase 1 | 기획 | QST, REQ, FN, IA, WBS, Dashboard |
| Phase 2 | 디자인 | DK, Layout, UI명세, 스타일가이드 |
| Phase 3 | 퍼블리싱 | HTML, CSS, JS, 반응형, 접근성 |
| Phase 4 | QA | 기능테스트, 접근성검사, 성능측정 |
| Phase 5 | 런칭 | 서버세팅, 배포, 모니터링, 고객인수 |

### 3. 작업 분해

#### 분해 규칙
- **작업 크기**: 1~5일 단위 (5일 초과 시 분할)
- **FN→작업 매핑**: 모든 FN이 1개 이상의 WBS 작업에 매핑
- **횡단 작업**: 다국어, 접근성, SEO는 별도 작업으로 분리

#### WBS 작업 필수 필드
| 필드 | 필수 | 설명 |
|------|------|------|
| WBS ID | 필수 | WBS-{Phase}.{순번} 형식 |
| 작업명 | 필수 | 동사형 (예: "상품 목록 퍼블리싱") |
| FN 근거 | 필수 | FN-### ID |
| 담당 | 필수 | 역할 (기획/디자인/퍼블/QA) |
| 기간 | 필수 | 일 단위 |
| 선행작업 | 필수 | WBS-_._  ID (없으면 "-") |
| 산출물 | 필수 | 결과물 파일/문서명 |
| Wave | 필수 | W1 / W2 / W3+ (병렬 실행 웨이브) |
| 우선순위 | 필수 | P0(크리티컬 패스) / P1(일반) |

#### 병렬 웨이브 분류 규칙

작업을 병렬 실행 가능 단위(Wave)로 분류합니다. Wave 간에는 의존성이 있고, Wave 내 작업은 동시 진행 가능합니다.

| Wave | 기준 | 예시 |
|------|------|------|
| W1 | 선행작업 없음, 즉시 착수 가능 | 기획 확정, 디자인 시안, 콘텐츠 수집 |
| W2 | W1 완료 후 착수 가능 | 퍼블리싱, 기능 개발, 콘텐츠 입력 |
| W3+ | W2 완료 후 착수 가능 | QA, 통합 테스트, 운영 이관 |

- **P0 (크리티컬 패스)**: 지연 시 전체 일정에 직접 영향. 간트 차트에서 굵게 표시
- **P1 (일반)**: 버퍼 내 지연 허용. 크리티컬 패스에 영향 없음
- Wave 판정: 선행작업 필드 기준 자동 분류 (선행작업 "-" → W1, W1 의존 → W2, W2 의존 → W3+)

### 4. 일정 산정

#### 공수 기준 (가이드)
| FN 복잡도 | 디자인 | 퍼블리싱 | QA |
|----------|--------|---------|-----|
| 높음 | 3~5일 | 3~5일 | 2~3일 |
| 중간 | 2~3일 | 2~3일 | 1~2일 |
| 낮음 | 1일 | 1일 | 0.5일 |

- 버퍼: 전체 일정의 10~15% 추가
- 병렬 가능 작업 식별 → 일정 단축

#### 역할×규모 참조 테이블 (인일 기준)

| 역할 | 소규모(5p 이하) | 중규모(6-15p) | 대규모(16p+) |
|------|:---:|:---:|:---:|
| 기획 | 19-38 | 38-76 | 76-114 |
| 디자인 | 15-29 | 29-57 | 57-86 |
| 퍼블리싱 | 13-35 | 35-70 | 70-105 |
| 개발 | 16-46 | 46-91 | 91-137 |

#### 업종별 보정 계수

_context.md 또는 QST에서 업종을 확인하여 적용합니다.

| 업종 | 계수 | 사유 |
|------|:---:|------|
| 금융 | ×1.15 | 보안·규제 |
| 공공 | ×1.20 | 접근성·행정절차 |
| 기업 | ×1.10 | 내부 승인 |
| 커머스 | ×1.10 | 결제·물류 연동 |
| 교육·의료 | ×1.15 | 규제·인증 |
| 글로벌 | ×1.30 | 다국어·시차 |

#### 3점 추정법 (대규모 또는 고위험)

```
공수 = (낙관치 + 4×최빈치 + 비관치) / 6
```

- 소규모: 단일 추정 (최빈치)
- 중규모: 범위 추정 (최빈치 ± 버퍼)
- 대규모: 3점 추정 필수

#### 버퍼 기준

| 조건 | 버퍼율 |
|------|:---:|
| 기본 | 10-15% |
| 고위험 (리스크 점수 6+) | 15-20% |
| 신기술 도입 | 추가 +30% |

### 4.5 리스크 평가

7대 리스크 유형별 확률 기준선:

| 유형 | 기본 확률 | 설명 |
|------|:---:|------|
| 일정 | 80% | 일정 지연 |
| 범위 | 75% | 범위 변경/확장 |
| 고객 | 60% | 의사결정 지연, 요구변경 |
| 리소스 | 50% | 인력 부족/이탈 |
| 기술 | 45% | 기술 난이도, 호환성 |
| 품질 | 40% | 결함, 성능 미달 |
| 커뮤니케이션 | 45% | 소통 오류 |

#### 3×3 영향 매트릭스

| | 낮음(1) | 중간(2) | 높음(3) |
|---|:---:|:---:|:---:|
| 높음(3) | 3 | 6 | **9** |
| 중간(2) | 2 | 4 | 6 |
| 낮음(1) | 1 | 2 | 3 |

- 점수 **6 이상**: 대응 전략 필수
- 점수 **3~5**: 모니터링 + 조건부 대응
- 점수 **1~2**: 수용

#### 대응 전략 4유형

| 전략 | 설명 |
|------|------|
| 회피 | 리스크 원인 제거 (범위 축소, 기술 변경) |
| 전가 | 제3자에게 이전 (외주, 보험) |
| 완화 | 확률/영향 감소 (프로토타입, 교육) |
| 수용 | 발생 시 대응 (버퍼 확보, 대기) |

#### 리스크 레지스터 출력

| # | 유형 | 리스크 | 확률(1-3) | 영향(1-3) | 점수 | 대응전략 | 조치 | 담당 |
|---|------|--------|:---:|:---:|:---:|----------|------|------|

- 최소 **5건** 이상 식별
- 7대 유형 중 **3개 이상** 커버
- 점수 **6 이상** 항목은 대응전략 + 조치 필수

### 5. 크리티컬 패스 식별
- 가장 긴 의존 경로 = 크리티컬 패스
- 크리티컬 패스 상의 작업은 지연 시 전체 일정에 영향
- 마일스톤별 게이트 조건 명시

## 컨텍스트 전파

### _context.md 읽기 (Step 0에서 수행)

`output/{프로젝트명}/_context.md` 존재 시 아래 필드를 로드하여 WBS 작성에 활용합니다.

| 필드 | 활용 |
|------|------|
| 프로젝트명 | WBS 문서 헤더 + 파일명에 동일 프로젝트명 사용 |
| 업종 | 공수 기준 보정 (업종별 복잡도 가중치) |
| FN 기능 수 / 복잡도 | Phase 2~3 작업 수 산정 근거 (높음→3~5일, 중간→2~3일, 낮음→1일) |
| FN 핵심 기능 (높음) | 크리티컬 패스 후보 식별 |
| IA 페이지 수 / Depth | 퍼블리싱 작업량 산정 근거 |
| REQ FR 우선순위 | Phase 분배 시 High FR 우선 배정 |

### _context.md 기록 (완료 시 필수)

WBS 생성 완료 후 `output/{프로젝트명}/_context.md`에 아래 블록을 **append** 합니다.

```markdown
## WBS
- 생성일: {YYYY-MM-DD}
- 총 작업 수: {n}건
- 총 기간: {n}주
- Phase별 분포: 기획{n} / 디자인{n} / 퍼블{n} / QA{n} / 런칭{n}
- 크리티컬 패스: {WBS-_._ → ... → WBS-_._} ({n}일)
- 버퍼 비율: {n}%
```

### Mini-PM 자가 점검 (3문항)

WBS 생성 완료 후, 결과 출력 전에 아래 3문항을 자가 점검합니다. 1개라도 No이면 해당 항목을 수정 후 출력합니다.

| # | 점검 문항 | 판정 |
|---|----------|------|
| PM-1 | FN→WBS 작업 매핑이 완결적인가? (orphan FN 0건) | Yes/No |
| PM-2 | 프로젝트명·업종이 _context.md와 일치하는가? | Yes/No/N/A |
| PM-3 | 기능 수 대비 작업 수가 합리적인가? (FN 1개당 평균 2~4 작업) | Yes/No |

## Step 0-F: 폴더 보장

산출물 저장 전에 아래 폴더 구조를 확인하고, 없으면 자동 생성합니다.

```
1. PROJECT.md 존재 여부로 프로젝트 루트 및 {프로젝트명} 결정 (미존재 → 사용자 확인)
2. output/{프로젝트명}/ 디렉토리 → 없으면 생성
3. output/{프로젝트명}/{YYYYMMDD}/ 디렉토리 → 없으면 생성 (YYYYMMDD = 오늘 날짜)
```

**출력 (필수)**:
```
[폴더 보장] output/{프로젝트명}/{YYYYMMDD}/: {존재/생성}
```

## 출력 마크다운 구조 (필수)

생성되는 MD 파일은 **반드시** 아래 H2 구조를 따라야 합니다. 구축/운영 모드 공통.

```markdown
# WBS 작업분해구조 — {프로젝트명}
> 작성일: {날짜} | 버전: v1.0 | 프로젝트 유형: {구축/운영}

## 1. 문서 개요
| 항목 | 내용 |
|------|------|
| 프로젝트명 | {값} |
| 입력 근거 | FN/IA 파일명 또는 "프롬프트 기반" |
| 총 작업 수 | {n}건 |
| 총 기간 | {n}주 ({시작일} ~ {종료일}) |

## 2. 마일스톤
| ID | 마일스톤 | 날짜 | 게이트 조건 | 상태 |
|----|---------|------|-----------|------|
| M1 | 기획 완료 | {날짜} | REQ+FN 승인 | 대기/진행/완료 |

## 3. 작업 분해 테이블
| WBS-#.# | Phase | 작업명 | 담당 | 선행 | 기간(일) | 시작 | 종료 | Wave | 우선순위 |
```
