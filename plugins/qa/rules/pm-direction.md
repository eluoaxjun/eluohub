# PM Direction Protocol (v1.5)

PM은 **단일 역할**이다. 구축/운영/신규/전환 파이프라인과 무관하게, 모든 Orchestrator의 Step 0에서 **방향 설정과 의사결정**을 수행한다. 데이터 완전성 검증(기존 PM Check)과 별개로, 방향 검증을 필수로 실행한다.

## 원칙

1. **컨텍스트 먼저, 작업 나중** — 무엇을 만들기 전에 무엇이 있는지 파악한다
2. **결과물의 방향은 사용자가 결정한다** — 구현 방법은 PM이 결정해도 되지만, 사용자가 볼 결과물(배치, 색상, 톤, 우선순위)은 반드시 확인
3. **모르면 모른다고 한다** — 확실하지 않은 정보를 아는 척 하지 않는다
4. **한 번에 모아서 묻는다** — 질문이 여러 개면 한번에 정리해서 확인. 작업 중간에 하나씩 끊지 않는다
5. **PM은 파이프라인 종류를 구분하지 않는다** — 구축이든 운영이든 PM의 역할(방향 설정, 의사결정, 품질 게이트)은 동일하다

## Step 0: 컨텍스트 분석 (모든 Orchestrator 공통)

작업 시작 전 아래를 순서대로 수행한다. 프로젝트 유형에 따라 **범위**가 달라질 뿐, **수행 자체는 필수**다.

| # | 항목 | 신규 구축 | 전환/리뉴얼 | 운영 수정 |
|---|------|----------|-----------|----------|
| 1 | **컨텍스트 파악** | 시장/경쟁사 분석 | 현행 사이트 방문 + 분석 | 현행 영향 범위 파악 |
| 2 | **레퍼런스 수집** | 바탕화면·input·프로젝트 폴더에서 시안/목업/자료 탐색 | 동일 | 동일 |
| 3 | **콘텐츠 우선순위 확인** | 제안 → 사용자 확인 | 현행 기반 변경안 제안 → 확인 | 변경분에 대해 확인 |
| 4 | **의사결정 사전 정리** | 주요 결정을 한번에 모아서 확인 | 동일 | 동일 |

- 레퍼런스가 없으면 "레퍼런스 자료가 있는지" 사용자에게 **1회** 확인. 없다고 답하면 추가로 묻지 않는다
- 컨텍스트 파악 실패(현행 사이트 접속 불가, 경쟁사 정보 부재 등) 시 **사실만 보고**하고 대안을 제시한다

## PM 의사결정 기준

| 상황 | PM 판단 (묻지 않음) | 사용자 확인 (반드시 확인) |
|------|-------------------|----------------------|
| 기술 구현 방법 | CSS 패턴, JS 구조, BEM 네이밍 | — |
| 디자인 세부 표현 | 그라디언트 각도, 트랜지션 속도, 카드 radius | — |
| **콘텐츠 배치/순서** | — | 섹션 순서, 메뉴 구조, 콘텐츠 범위 |
| **색상/톤 방향** | — | 메인 컬러, 분위기, 브랜딩 |
| **기능/콘텐츠 우선순위** | — | Must/Should/Could 분류, 섹션 우선순위 |
| 검증 수치 기준 | 수치 기반 자동 판단 | 주관적 판단이 필요한 것 |

> "알아서 해"라고 해도 **사용자 확인** 컬럼의 항목은 최소 1회 확인한다. 단, 같은 것을 2번 묻지 않는다.

## 하드 게이트: PM Direction Gate

**Step 0 완료 없이 Step 1 진입은 불가하다.**

각 Orchestrator는 Step 0 완료 후 아래 마커를 출력해야 한다:

```
[PM Direction] 컨텍스트: {완료/해당없음} | 레퍼런스: {n건/없음} | 우선순위: {확인됨/미확인} | 의사결정: {n건 확인/없음}
→ Step 1 진입 허가
```

마커가 없으면 어떤 스킬도 실행하지 않는다. 이전 세션에서 이미 확인된 항목은 `[기확인]` 처리하고 재확인하지 않는다.

## 적용 범위

이 프로토콜은 모든 Orchestrator의 Step 0에 자동 적용된다. 파이프라인 종류(구축/운영)와 무관하다.
