# 기능정의서 (FN)

> 각 기능이 화면에서 어떻게 보이고, 어떻게 동작하는지 상세하게 설명해줍니다.

---

## 한 줄 요약

각 기능의 **입력-처리-출력**과 정상·예외·에러 시나리오를 상세 명세한 문서입니다.

---

## 이런 상황에서 써요

- 요구사항(REQ)을 디자인·개발이 바로 착수할 수 있는 수준으로 구체화해야 할 때
- 화면 구성 요소와 동작 흐름을 명확히 정의해야 할 때
- QA 검증 기준을 기능 단위로 수립해야 할 때

---

## 사용법

채팅창에 자연스럽게 말하면 됩니다:

```
"기능 상세 정의해줘"
"기능 명세 만들어줘"
"각 기능이 어떻게 동작하는지 정리해줘"
```

---

## 어떤 결과가 나오나요?

**파일**: `output/planning/FN_{프로젝트코드}_v1.md`

각 기능(FN)마다 아래 내용이 포함됩니다:

| 항목 | 설명 |
|------|------|
| 화면 구성 | 해당 기능의 UI 요소와 배치 |
| 입력 | 사용자 액션 및 입력 데이터 |
| 처리 | 시스템 내부 로직과 데이터 흐름 |
| 출력 | 처리 결과와 화면 반영 내용 |
| 검증 기준 | 정상·예외·에러 각 시나리오별 기대 결과 |

---

## 산출물 예시

### 기능 한 건 (FN-001: 도서 검색/목록)

```markdown
### FN-001: 도서 검색/필터

- **FR 근거**: FR-001
- **화면 구성**:
  - 상단: 장르 필터 바 (소설/에세이/자기계발/만화)
  - 중단: 카드형 목록 (표지, 도서명, 저자, 가격)
  - 하단: 페이지네이션 (10건/페이지)
- **입력**: 장르 선택, 정렬 기준, 페이지 번호
- **처리**: 선택된 장르로 필터링 → 정렬 → 페이지네이션
- **출력**: 필터링된 도서 카드 목록 + 총 건수
- **검증 기준**:
  - 정상: 필터 선택 시 해당 장르만 표시
  - 예외: 결과 0건이면 "도서가 없습니다" 안내
  - 에러: 데이터 로딩 실패 → 재시도 버튼 표시
```

### 기능 목록 요약표

```markdown
| FN ID | 기능명 | FR 근거 | 우선순위 |
|-------|--------|---------|---------|
| FN-001 | 도서 검색/필터 | FR-001 | 상 |
| FN-002 | 도서 상세 보기 | FR-002 | 상 |
| FN-003 | 추천 도서 목록 | FR-003 | 상 |
| ... | ... | ... | ... |
```

### 의존관계 맵

```markdown
FN-010 (메인) ──→ FN-001, FN-003, FN-005 (콘텐츠 소스)
FN-008 (검색) ──→ FN-001~FN-007 (검색 대상)
FN-009 (장바구니) ──→ FN-002, FN-003 (상품 연동)
```

---

## 이 문서를 읽는 사람은?

- **디자이너**: 화면에 뭘 넣어야 하는지 (화면 구성)
- **개발자**: 어떻게 만들어야 하는지 (입력→처리→출력)
- **QA 담당자**: 뭘 테스트해야 하는지 (검증 기준)

---

## 팁 & 주의사항

### 독립 모드 (REQ 없이 실행할 때)

REQ 산출물 없이 FN만 단독으로 실행하면 자동으로 **독립 모드**로 동작합니다.

- `FR 근거: 생략 (독립 모드)`로 처리되며, FR-### 역방향 추적은 하지 않습니다
- 기능 수가 파이프라인 모드보다 적게 나올 수 있습니다 — 프롬프트에 도메인과 주요 기능을 구체적으로 명시하면 더 많은 기능이 도출됩니다

```
# 권장 프롬프트 (구체적 명시)
"엘루오 허브 기능정의서 만들어줘.
 스킬 다운로드, 설치 관리, 카테고리 검색, 마이페이지 기능 포함."

# 비권장 (추상적)
"기능정의서 만들어줘."
```

### 파이프라인 모드 (REQ 연계)

선행 REQ 파일이 `output/planning/` 또는 `input/`에 있으면 자동으로 파이프라인 모드로 전환됩니다.

- FR-### → FN-### 추적성이 자동 적용됩니다
- REQ에 정의된 모든 FR을 FN으로 전환하므로 기능 누락 위험이 낮습니다
